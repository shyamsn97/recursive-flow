# Recursive Agent Optimization: Implementation Plan

This note sketches how to implement Recursive Agent Optimization (RAO) on top of
`rlmflow`.

Sources:

- RAO project page: <https://apga.github.io/RAO/>
- Local `rlmflow` architecture docs: `README.md`, `docs/control.md`,
  `docs/internals.md`, `docs/node_model.md`, and `docs/observability.md`.

## Short Answer

RAO is a natural fit for this repo, but it should be implemented as a
first-class `rflow.rao` library module around `RecursiveFlow`, not as a rewrite
of the runtime.

`rlmflow` already has most of the inference substrate RAO assumes:

- one recursive agent policy used at every tree depth;
- a Python REPL action loop;
- `launch_subagents([...])` for recursive delegation;
- a durable tree of per-agent trajectories via `Graph.children`;
- per-agent task payloads through `Graph.context` / `CONTEXT`;
- depth and iteration controls through `FlowConfig`;
- workspaces for replay, inspection, forking, and dataset export.

The missing RAO pieces are training-specific:

- benchmark task samplers;
- per-node reward functions;
- rollout grouping;
- delegation-aware reward aggregation;
- advantage computation;
- depth-level inverse-frequency weighting;
- policy update/export integration.

So the right implementation boundary is:

```text
rflow runtime
  RecursiveFlow
  Graph / Node
  Workspace
  launch_subagents

RAO research layer
  task samplers
  rollout collectors
  reward functions
  advantage / weighting
  training example exporter
  trainer adapter
```

## What RAO Needs

RAO trains a single shared policy to use recursive delegation as an
inference-time primitive. The policy runs at the root and at every child node.
During a rollout, each agent instance can solve locally, spawn child agents, wait
for child results, and synthesize an answer.

The RAO project page describes three important ideas:

- **recursive agent inference**: agents can launch recursive copies of the same
  policy on sub-tasks;
- **local node reward with delegation bonus**: each node receives reward for its
  own task success plus a bonus for successful immediate children;
- **depth-weighted multi-task optimization**: trajectories from all tree depths
  train the same policy, but deeper nodes are reweighted so they do not dominate
  updates just because they are more numerous.

That maps cleanly onto a `Graph`: each `Graph` object is one agent trajectory,
and `graph.children` is the recursive execution tree.

## Core Data Model

Do not add RAO fields directly to `Graph` unless they are durable run state
needed for normal inference. Rewards, advantages, and trainer metadata are
research annotations over a graph, not core graph state.

Use separate dataclasses:

```python
from dataclasses import dataclass, field
from typing import Any

import rflow


@dataclass(frozen=True)
class TaskSpec:
    id: str
    query: str
    context: str = ""
    metadata: dict[str, Any] = field(default_factory=dict)


@dataclass(frozen=True)
class NodeScore:
    agent_id: str
    local_score: float
    child_score: float
    reward: float
    metadata: dict[str, Any] = field(default_factory=dict)


@dataclass(frozen=True)
class TrajectoryExample:
    task_id: str
    rollout_id: str
    agent_id: str
    depth: int
    messages: list[dict[str, Any]]
    actions: list[dict[str, Any]]
    reward: float
    advantage: float
    weight: float
    metadata: dict[str, Any] = field(default_factory=dict)
```

`TaskSpec` is benchmark input. `NodeScore` is reward output. `TrajectoryExample`
is what gets exported to a trainer.

## Reward Interface

RAO needs local task reward at every node. That means the reward function must be
able to score not only the root task, but also child tasks generated by the
policy.

The smallest useful interface:

```python
from typing import Protocol


class RewardFn(Protocol):
    def score(self, task: TaskSpec, graph: rflow.Graph) -> float:
        """Return this agent node's local success/proxy score."""
```

For benchmark tasks where child tasks have no gold label, use proxy reward:

- exact evaluator if the child context includes a verifiable target;
- LLM judge for decomposition quality or evidence quality;
- parent-consistency score if the child output can be checked against the
  parent's final result;
- zero local child score at first, then only train root success until a better
  evaluator exists.

The important constraint: reward code should be benchmark-owned. The engine
should not know what "success" means.

## RAO Reward Computation

For each agent graph `X`, compute:

```text
reward(X) = local_score(X) + lambda * mean(local_score(child) for child in X.children)
```

Use the immediate-child success rate, not raw child count, so the model is not
rewarded for spawning more children.

Implementation sketch:

```python
def score_tree(
    graph: rflow.Graph,
    task_for_agent: dict[str, TaskSpec],
    reward_fn: RewardFn,
    *,
    delegation_bonus: float,
) -> dict[str, NodeScore]:
    local = {
        agent.agent_id: reward_fn.score(task_for_agent[agent.agent_id], agent)
        for agent in graph.walk()
    }

    scores: dict[str, NodeScore] = {}
    for agent in graph.walk():
        child_scores = [local[child.agent_id] for child in agent.children.values()]
        child_score = sum(child_scores) / len(child_scores) if child_scores else 0.0
        reward = local[agent.agent_id] + delegation_bonus * child_score
        scores[agent.agent_id] = NodeScore(
            agent_id=agent.agent_id,
            local_score=local[agent.agent_id],
            child_score=child_score,
            reward=reward,
        )
    return scores
```

The missing piece is `task_for_agent`. Today child `Graph.query` and
`Graph.context` hold enough information for many benchmarks. For stricter RAO,
we may want to persist a normalized child `TaskSpec` whenever a sub-agent is
spawned.

## Advantage Computation

RAO groups multiple root rollouts for the same task and computes a leave-one-out
baseline from root rewards. Then that root-rollout advantage is applied to every
trajectory in the rollout tree.

```python
def leave_one_out_advantages(root_rewards: list[float]) -> list[float]:
    if len(root_rewards) <= 1:
        return [0.0 for _ in root_rewards]
    total = sum(root_rewards)
    return [
        reward - (total - reward) / (len(root_rewards) - 1)
        for reward in root_rewards
    ]
```

Depth weighting should be applied after advantage computation:

```python
def depth_weights(graphs: list[rflow.Graph]) -> dict[int, float]:
    counts: dict[int, int] = {}
    for graph in graphs:
        for agent in graph.walk():
            counts[agent.depth] = counts.get(agent.depth, 0) + 1

    if not counts:
        return {}
    mean_count = sum(counts.values()) / len(counts)
    return {depth: mean_count / count for depth, count in counts.items()}
```

This preserves the contribution of deeper trajectories without letting depth 2+
dominate just because recursion fans out.

## Rollout Collection

Add RAO as a first-class library module. It should be separate from the core
runtime loop, but still live under `rflow/` because rollout collection, reward
aggregation, and training export are reusable library features:

```text
rflow/rao/
  __init__.py
  tasks.py          # TaskSpec, TaskSampler
  rewards.py        # RewardFn, score_tree
  collect.py        # RAORolloutCollector
  advantages.py     # LOO baseline, depth weights
  export.py         # trainer example export
  trainers.py       # adapter interfaces
```

Keep RAO out of `RecursiveFlow.step(...)` and the graph data model; do not keep
it out of the package. The clean boundary is a sibling package: `rflow/rao`
consumes `RecursiveFlow`, `Graph`, `Workspace`, transcripts, and contexts.
Example scripts should be thin entrypoints that import from `rflow.rao`.

Collector shape:

```python
class RAORolloutCollector:
    def __init__(
        self,
        *,
        engine_factory,
        workspace_factory,
        reward_fn: RewardFn,
        rollouts_per_task: int,
        delegation_bonus: float = 0.4,
    ) -> None:
        ...

    def collect(self, tasks: list[TaskSpec]) -> list[TrajectoryExample]:
        ...
```

For each task:

1. Create `G` isolated workspaces.
2. Create `G` `RecursiveFlow` engines using the same policy.
3. Run `engine.start(task.query, context=task.context)`.
4. Step each graph until done, max iterations, or error.
5. Score each rollout tree.
6. Compute leave-one-out root advantages.
7. Export one training example per agent trajectory.

This can run sequentially first. Parallel collection can come later using the
same scheduler pool or an external job runner.

## Exporting Training Examples

Training should use the actual LLM transcript, not just typed graph nodes.

`Graph.nodes` is perfect for observability and reward analysis. For policy
updates, the trainer needs the prompt/messages and chosen action text. The
workspace transcript files already preserve the message-level view. The exporter
should therefore prefer:

```text
workspace/session/<agent>/transcript.json
```

and fall back to reconstructing from `Graph.nodes` only for offline tests.

A minimal export record:

```json
{
  "task_id": "textcraft-123",
  "rollout_id": "textcraft-123/g0",
  "agent_id": "root.child_2",
  "depth": 1,
  "messages": [],
  "actions": [],
  "reward": 0.8,
  "advantage": 0.3,
  "weight": 0.5,
  "metadata": {
    "local_score": 0.6,
    "child_score": 0.5,
    "max_depth": 3
  }
}
```

The trainer adapter can decide how to convert this into GRPO, PPO, DPO-style
preference pairs, or an external provider's expected format.

## Trainer Boundary

Do not make `rlmflow` own model training at first. Different users will
train through different systems: TRL, OpenRLHF, verl, Tinker, internal APIs, or
provider-specific fine-tuning.

Define a narrow adapter:

```python
class TrainerAdapter(Protocol):
    def update(self, examples: list[TrajectoryExample]) -> dict[str, Any]:
        """Run one policy update and return metrics."""
```

For a first research prototype, implement `JsonlTrainerAdapter` that only writes
examples:

```text
runs/rao/textcraft/updates/000001/examples.jsonl
runs/rao/textcraft/updates/000001/metrics.json
```

That lets us validate rollout collection, rewards, advantages, and depth
weighting before coupling to a specific training stack.

## Benchmarks To Start With

### 1. Synthetic Crafting

This should be the first RAO benchmark because it matches the recursive shape
cleanly:

- tasks are generated from a hidden recipe DAG/tree;
- root query asks for a target item;
- child tasks can ask for sub-components;
- success is exact and cheap to score;
- depth difficulty is controllable.

Start with a simplified TextCraft-style environment:

```python
TaskSpec(
    id="craft-0001",
    query="Craft a beehive. Return the ordered recipe plan.",
    context=json.dumps({
        "inventory": ["plank", "string", "honeycomb"],
        "recipes": {...},
        "target": "beehive",
    }),
)
```

Reward can be exact-match or graph-plan validity.

### 2. Long-Context QA

This maps to the repo's existing needle/haystack examples:

- root context is too large for comfortable single-agent solving;
- the policy can chunk and delegate;
- child nodes answer evidence-local questions;
- root aggregates.

Reward is root exact answer first. Child local reward can be added later by
checking whether child outputs contain the needed evidence.

### 3. Deep Research

This should wait until the collection/export stack is stable. Rewards are harder
and likely require LLM judging or a task-specific verifier.

## Required Engine Additions

Keep these small:

1. **Spawn metadata capture**

   Persist enough information to reconstruct each child `TaskSpec`:

   ```text
   agent_id
   parent_agent_id
   query
   context
   model
   output_schema
   spawn node id
   ```

   Much of this is already in `Graph`; the main question is whether the exact
   launch spec should be stored as node metadata for cleaner reward analysis.

2. **Rollout terminal status**

   The collector needs to distinguish:

   ```text
   success
   normal done with wrong answer
   max-iteration stop
   runtime error
   validation error
   ```

   This can be research-layer metadata derived from the final node at first.

3. **Budget controls**

   RAO exploration can explode. The collector should enforce:

   ```text
   max_depth
   max_iterations
   max_children_per_agent
   max_total_agents
   max_wall_time
   ```

   Some already exist; `max_children_per_agent` and `max_total_agents` may need
   explicit scheduler checks.

4. **Transcript export API**

   Add a stable helper instead of making RAO code know workspace internals:

   ```python
   workspace.session.read_transcript(agent_id)
   ```

   This already exists on the session interface; RAO export should use it.

## What Not To Do

Do not:

- bake RAO rewards into `RecursiveFlow.step`;
- add reward/advantage fields to every `Node`;
- make `Workspace` understand training updates;
- train directly from rendered graph visualizations;
- require a specific model trainer before the rollout data pipeline works;
- reward raw delegation count.

The engine should stay an execution substrate. RAO should be a consumer of
graphs, transcripts, contexts, and workspace runs.

## Milestones

### Milestone 1: Offline RAO Dataset Export

Goal: collect recursive rollouts and export scored trajectory examples, without
training.

Deliverables:

- `TaskSpec`, `RewardFn`, `NodeScore`, `TrajectoryExample`;
- `score_tree(...)`;
- leave-one-out baseline;
- depth weighting;
- JSONL exporter;
- unit tests on synthetic hand-built graphs;
- one tiny crafting benchmark with exact reward.

Success criterion:

```text
python examples/research/rao_collect_crafting.py
```

writes a JSONL file where every agent trajectory has reward, advantage, and
depth weight.

### Milestone 2: Policy-Update Adapter

Goal: connect exported examples to one trainer without making the repo depend on
that trainer.

Deliverables:

- `TrainerAdapter` protocol;
- `JsonlTrainerAdapter`;
- one optional TRL/GRPO adapter in an examples folder or extra dependency;
- update metrics saved as artifacts.

Success criterion:

One update step can consume examples and produce metrics, even if the default
adapter only writes JSONL.

### Milestone 3: End-To-End Synthetic RAO

Goal: show that recursive policies improve on a small controlled task.

Deliverables:

- synthetic crafting task generator;
- train/eval split by difficulty depth;
- single-agent baseline using `max_depth=0`;
- recursive agent treatment using `max_depth>0`;
- plots for success rate, depth distribution, steps, and wall-clock time.

Success criterion:

The recursive treatment learns to delegate on deeper tasks and outperforms the
single-agent baseline on held-out deeper tasks.

### Milestone 4: Long-Context RAO

Goal: adapt the needle/haystack or transcript QA examples into a benchmark where
recursion extends effective context.

Deliverables:

- long-context task sampler;
- exact root reward;
- optional evidence-local child reward;
- comparison across context lengths.

Success criterion:

The recursive policy maintains reward as context grows beyond the comfortable
single-agent setting.

## Open Questions

- Should child launch specs be stored as explicit typed nodes, or is current
  `Graph.query` / `Graph.context` enough?
- Do we want `max_children_per_agent` and `max_total_agents` in `FlowConfig`?
- Which trainer stack should be the first real adapter?
- Should rollout collection use one workspace per rollout or one workspace per
  task with child run directories?
- How should failed runtime executions be represented in the training example:
  negative reward, zero reward, or filtered?
- For child nodes without gold labels, what proxy reward is least misleading?

## Recommendation

Start with offline dataset export and a synthetic crafting benchmark.

That gives us the RAO core without premature trainer coupling:

```text
rlmflow already gives us recursive inference trees.
RAO first needs reliable rollout scoring and export.
Only after that should we wire in policy updates.
```
